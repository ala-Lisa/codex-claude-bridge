# Codex 额度节流设计

## 范围

在保留 DeepSeek 自动修复、Claude 实时可见性和 Codex 独立验收的前提下，减少重复的 Codex 审查消耗。

本次修改只适用于本地 `codex-claude-bridge` Skill，不修改 Shell 配置、模型配置、目标仓库或 Claude/Codex CLI。

## 已确认的当前行为

- Claude 持续输出 `stream-json`；桥接器会立即脱敏并转发这些事件。
- 浏览器监控页通过 SSE 接收桥接器事件，不会调用模型。
- `MonitorSampler` 当前每秒读取一次本地 Git、配置、内存和进程状态；该采样不会调用 Codex。
- 浏览器遥测每 10 秒记录一次当前快照。
- Codex 只在一次 Claude 调用完成后启动。
- 后续 Codex 审查会复用原 Codex 会话，但当前提示会重复发送稳定的任务上下文和较大的仓库 diff。
- Claude 报告的测试失败目前不会在 Codex 启动前经过桥接器独立执行的命令闸门。

## 目标

1. 保留即时、脱敏的 Claude 文本和工具活动显示。
2. 确定性测试失败时，不调用 Codex，直接返回同一个 Claude/DeepSeek 会话。
3. 只有通过全部已批准本地验证命令的候选版本才能进入 Codex 审查。
4. Codex 返回的每个 `FAIL` 都自动返回同一个 Claude/DeepSeek 会话。
5. 要求 Codex 在每次审查中一次性报告已发现的全部范围内阻塞问题。
6. 后续审查不重复发送稳定任务上下文和大段 diff。
7. 保留 Codex 独立只读检查、`NEEDS_INPUT`、并发锁、日志脱敏、超时、脏工作区保护和有限安全停止机制。

## 非目标

- 不把 Claude 文本改成每 5 秒采样；Claude 文本仍然由事件实时驱动。
- 不声称浏览器或本地采样会消耗 Codex 额度。
- 不允许用 Claude 的文字报告替代桥接器独立执行验证命令。
- 已批准检查通过后，不跳过 Codex 审查。
- 不引入更便宜的审查模型，不静默更改所选 Codex 模型，也不降低用户选择的推理强度。
- 不执行未经批准的 Shell 文本，不新增依赖。
- 不承诺没有边界的无限循环；正常流程无需人工介入，但异常运行仍受已配置的安全上限约束。

## 设计

### 1. 监控频率

把 `MonitorSampler` 默认间隔从 1 秒改为 5 秒。该修改只影响本地 Git、配置、内存和进程指标。

以下行为保持不变：

- Claude 和 Codex 的流事件一到达，就立即转发到终端和 SSE 监控页。
- 浏览器遥测仍然每 10 秒向折线历史追加一个数据点。

因此，模型输出继续实时显示，同时减少本地文件系统和 `/proc` 的读取频率。

### 2. 已批准验证清单

新增可选运行参数：

```text
--verification-file /absolute/path/to/verification.json
```

由 Codex 编写实施计划时，在任务和计划输入文件旁生成不含凭据的验证清单。用户一次性批准任务、计划和验证清单后，桥接器才可启动。新的 Skill 使用说明要求：只要已批准计划包含可执行检查，就必须提供验证清单；旧任务未提供该文件时，保持原有的直接审查行为以兼容现有流程。

验证清单使用以下精确 JSON 结构：

```json
{
  "version": 1,
  "commands": [
    [".venv/bin/python", "-m", "pytest", "-q"],
    [".venv/bin/python", "-m", "pytest", "-q", "-W", "error"],
    [".venv/bin/python", "-m", "pip", "check"],
    ["git", "diff", "--check"]
  ]
}
```

在任一模型启动前验证该文件：

- 必须是 UTF-8 JSON 对象，且只能包含 `version` 和 `commands`；
- `version` 必须是整数 `1`，不能是 `bool`；
- 命令数量为 1–20 条；
- 每条命令必须是非空列表，包含 1–64 个非空字符串；
- 参数不得包含 NUL、换行、疑似凭据值或 Shell 控制运算符；
- 文件大小不得超过 64 KiB；
- 不得把参数数组拼装成 Shell 命令字符串。

桥接器必须原样执行用户已批准的参数数组，不得从自然语言中推断、改写、追加或发现验证命令。

在 `state.json` 中保存 SHA-256 摘要和已脱敏的解析后参数数组。恢复任务时未再次提供文件，则使用状态中已保存的命令；再次提供的文件摘要与原摘要不一致时，必须在任一模型启动前停止。不得保存未脱敏的清单原文。

### 3. 桥接器独立验证闸门

每次 Claude 子进程成功结束后，状态转为 `VERIFYING`，并按顺序执行已批准命令。执行要求：

- 使用 `subprocess.Popen` 参数列表；
- 使用 `shell=False`；
- 工作目录固定为目标仓库；
- 并发排空 stdout 和 stderr，避免管道死锁；
- 每条命令使用正整数 `--verification-timeout`，默认 1,800 秒；
- 每个输出流最多保留 20,000 个脱敏字符；
- 显示或持久化之前必须脱敏。

每次实施尝试保存一个脱敏验证结果文件。遇到第一条非零退出命令即停止本轮验证，因为后续检查可能依赖前序检查成功。

验证命令失败时：

1. 不启动 Codex；
2. 生成简洁修复指令，包含命令标签、退出码和有限长度的脱敏 stdout/stderr；
3. 将修复指令发送回同一个 Claude 会话；
4. Claude 返回后，重新执行完整验证清单。

Claude 可以在实施过程中自行运行检查，但只有桥接器独立执行全部命令成功后，才允许启动 Codex 审查。

### 4. 自动 Codex 修复循环

全部已批准命令通过后，以只读模式启动 Codex。

Codex 返回值的处理规则：

- `PASS`：正常完成任务。
- `NEEDS_INPUT`：保存 `AWAITING_INPUT`，释放锁并正常暂停。
- `FAIL`：把完整的定向修复指令自动发送回同一个 Claude 会话；Claude 修复后，必须再次经过完整验证闸门，才能重新进入 Codex 审查。

加强 Codex 审查指令：Codex 必须检查实际仓库，并一次性报告本轮发现的全部范围内阻塞问题，不得故意只报告第一个问题；同时不得为了增加问题数量而加入推测性或范围外工作。

### 5. 分离的有限计数器

一次“实施尝试”代表一次 Claude 子进程调用；只有验证闸门全部通过后才会产生一次 Codex 审查。两者必须在状态和 HUD 中分别计数。

新增以下正整数运行上限：

```text
--max-implementation-attempts 12
--max-codex-reviews 8
```

保留 `--max-rounds`，作为 `--max-codex-reviews` 的兼容别名。同时提供且数值不同必须报错。新的 Skill 示例统一使用含义明确的新参数。

恢复任务时，未提供的上限沿用状态中保存的值；显式提供但与状态冲突时，必须在任一模型启动前停止。达到任一上限时，进入受控 `STOPPED`，且不得再启动新的子进程。这两个上限只是异常安全停止，不是需要用户逐批批准的机制。

旧的进行中状态如果只有 `round` 和 `max_rounds`，则把保存的 `max_rounds` 原值迁移为 Codex 审查上限，并把实施尝试上限初始化为 12 与该值两倍中的较大值。必须在启动任一模型前保存迁移后的计数器，且不得在迁移时降低已有任务的上限。

HUD 同时显示两个计数器，例如：

```text
DeepSeek 实施 3 / 12
Codex 审查 1 / 8
```

### 6. 分级审查提示

第一次 Codex 审查必须具备完整、自包含的上下文，包括：

- 原始任务；
- 已批准计划和成功条件；
- Claude 实施报告；
- 桥接器独立验证结果；
- `git status --short`；
- `git diff --stat`；
- 有长度上限的 diff 摘要；
- 要求 Codex 独立检查实际仓库的指令。

默认 diff 摘要上限从 120,000 个字符降为 20,000 个字符。保留 `--max-diff-chars` 供具体任务显式覆盖。

后续审查复用原 Codex 会话，只发送：

- 上次审查的剩余问题和定向修复指令；
- 最新 Claude 修复报告；
- 最新桥接器独立验证结果；
- 当前 `git status --short`、`git diff --stat` 和变更文件路径；
- 要求重新读取实际文件和相关证据的明确指令。

后续审查不得重复发送原始任务、已批准计划或 diff 正文。后续审查时如果缺少 Codex 会话 ID，必须受控停止，不得把不完整上下文发送给新会话。

### 7. 安全和状态

- 提示、事件、JSONL、终端输出、验证命令输出和模型输出在显示或保存前继续统一脱敏。
- 不记录环境变量或凭据。
- 保留带所有者令牌的并发锁语义。
- 保留现有 `NEEDS_INPUT` 和用户回答文件验证行为。
- 保留仅作用于 Claude 子进程的自动压缩环境变量。
- 自动修复和用户回答恢复时，继续复用 Claude/Codex 会话 ID。
- 不使用 `bypassPermissions`，不提交、不推送、不修改 Git 历史。

## 离线验证要求

只能使用临时仓库和模拟 Claude/Codex 可执行文件。

必须执行以下检查：

1. 修改后的 Python 文件全部通过语法编译检查。
2. `quick_validate.py` 输出 `Skill is valid!`。
3. 现有桥接器和监控页测试全部通过。
4. 本地采样默认间隔为 5 秒。
5. Claude 局部文本和工具事件继续即时显示，不等待 5 秒采样周期。
6. 浏览器图表历史仍使用 10 秒间隔。
7. 验证清单的格式错误、超限、不安全、疑似凭据和恢复冲突输入，必须在任一模型启动前被拒绝。
8. 验证命令必须使用参数数组和 `shell=False`，同时排空两个输出流且不死锁。
9. 验证命令失败时，再次调用 Claude，但 Codex 调用次数为 0。
10. 后续验证成功时，Codex 调用次数为 1。
11. “验证失败 → Claude 修复 → 验证通过 → Codex PASS”必须自动完成，且只调用一次 Codex。
12. “Codex FAIL → Claude 修复 → 验证通过 → Codex PASS”必须自动完成，调用两次 Codex，并复用两个会话。
13. 本地验证持续失败时，在实施尝试上限停止，且不得调用 Codex。
14. Codex 持续失败时，在 Codex 审查上限停止，且不得多启动一次 Claude 或 Codex。
15. 第一次 Codex 提示必须包含原始任务、计划、桥接器验证证据和有限长度 diff 摘要。
16. 后续 Codex 提示不得重复任务、计划和 diff 正文，但必须包含上次问题、当前报告、验证证据、状态、统计和变更路径。
17. `NEEDS_INPUT` 仍会暂停，且不会再次调用 Claude。
18. 终端、事件、提示、验证结果和保存输出中均不得泄露敏感值。
19. 第二个进程不得替换状态，也不得删除第一个进程的锁。
20. 继续兼容使用 `--max-rounds` 的现有调用方式。
21. 最终文件列表不得包含范围外修改。

所有检查均不得调用真实 Claude、DeepSeek、Codex 或其他网络 API。
